   400‚Üí### 1.7 `/tmp/real3/src/lib/shared-token.ts`
   401‚Üí
   402‚Üí**Purpose:** Manage a shared Google JWT pool so users without their own Google OAuth can still hit Bearer-protected endpoints.
   403‚Üí
   404‚Üí**Storage:** Prisma `AppSetting` table, encrypted-at-rest (per docstring).
   405‚Üí- Key `shared_google_jwt` ‚Üí the JWT string.
   406‚Üí- Key `shared_google_jwt_expiry` ‚Üí ISO timestamp.
   407‚Üí
   408‚Üí**Functions:**
   409‚Üí- `getSharedToken()` ‚Äî returns the JWT string, or null if missing/expired/shorter than 20 chars.
   410‚Üí- `setSharedToken(token, expiryDays=30)` ‚Äî upserts both keys.

  1127‚Üí
  1128‚Üí**Conditional / OPTIONAL on the main transfer path** (the working "bypass mode"):
  1129‚Üí
  1130‚ÜíFrom `/tmp/real3/src/app/api/fantasy/transfer/route.ts` lines 342‚Äì351 and 433‚Äì436:
  1131‚Üí```ts

  1134‚Üí  // Don't return ‚Äî fall through and try the backend anyway
  1135‚Üí}
  1136‚Üí// ... later:
  1137‚Üíconst headers: Record<string, string> = { "Content-Type": "application/json" };
  1138‚Üíif (bearerToken && bearerToken.length >= 20) {

  1158‚Üí## 4. How `authToken` is obtained (OTP flow) and its exact format
  1159‚Üí
  1160‚Üí### 4.1 Flow
  1161‚Üí
  1162‚Üí1. User opens OTP dialog, enters 10-digit mobile number.
  1163‚Üí2. Frontend POSTs `/api/fantasy/send-otp` with `{ fantasyApp, mobileNumber }`.
  1164‚Üí3. Backend `tgsoftware-api.online/api/fantasy/send-otp` returns `{ status:"success", data:{ state?, reasonCode?, challenge?, login_auth_token?, userId? }, message }`.
  1165‚Üí4. Frontend saves platform-specific fields (state for dream11, reasonCode/challenge for my11circle, login_auth_token for myteam11, userId for vision11).
  1166‚Üí5. User enters 6-digit OTP.
  1167‚Üí6. Frontend POSTs `/api/fantasy/verify-otp` with `{ fantasyApp, mobileNumber, verificationCode, [platform-specific fields] }`.
  1168‚Üí7. Backend returns `{ status:"success", data: <possibly-AES-encrypted> }`.
  1169‚Üí8. The Next.js verify-otp route:
  1170‚Üí   - Decrypts `data.data` if string.
  1171‚Üí   - Recursively searches for a token in keys: `token`, `authToken`, `access_token`, `accessToken`.
  1172‚Üí   - Returns `{ status:"success", token: <RAW FULL token>, accessToken: <extracted JWT>, data, my11circleChallenge?, my11circleUserId? }`.
  1173‚Üí9. Frontend stores `token` (the RAW token, NOT extracted `accessToken`) as `FantasyAccount.authToken` in localStorage `tg_multi_accounts`.
  1174‚Üí
  1175‚Üí### 4.2 Exact authToken format
  1176‚Üí
  1177‚ÜíThe authToken stored and sent to the transfer endpoint is the **RAW token string returned by the backend**, which may be in either of two forms:
  1178‚Üí
  1179‚Üí**(a) Plain JWT string:**
  1180‚Üí```
  1181‚ÜíeyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOi...
  1182‚Üí```

  1253‚Üí## 6. Success / failure detection logic
  1254‚Üí
  1255‚Üí### 6.1 Backend transfer success criterion (transfer/route.ts `isTransferSuccess`)
  1256‚Üí
  1257‚Üí**SUCCESS = `data.status === "success"` (string equality, case-sensitive).**
  1258‚Üí
  1259‚ÜíEverything else is a FAILURE ‚Äî including:
  1260‚Üí- HTTP 200 + `{ status:"fail", message:"..." }`
  1261‚Üí- HTTP 201 + `{ status:"fail", message:"Something Went Wrong!" }` (this was previously misclassified as "tentative success"; the comment block explicitly documents the fix)
  1262‚Üí- HTTP 4xx/5xx with any body
  1263‚Üí
  1264‚ÜíThe route logs `confidence: 'definite'` for both success and failure (no 'tentative' is ever returned by the current code ‚Äî that branch was removed).
  1265‚Üí
  1266‚Üí### 6.2 Frontend success criterion (TeamTransferScreen.tsx executeTransferBatch)
  1267‚Üí
  1268‚Üí```ts
  1269‚Üíconst data = await res.json();
  1270‚Üíif (data.status === "success") {
  1271‚Üí  successCount++;
  1272‚Üí  return { success: true, index: i };
  1273‚Üí}
  1274‚Üí```
  1275‚ÜíIdentical to backend: only `data.status === "success"` counts.
  1276‚Üí
  1277‚Üí### 6.3 Token-expiry detection (transfer/route.ts `isConfirmedTokenExpiry`)
  1278‚Üí
  1279‚ÜíReturns `true` if any of these substrings appear in the error message (case-insensitive):
  1280‚Üí- `invalid token`, `token expired`, `token is expired`, `access token expired`
  1281‚Üí- `jwt expired`, `jwt malformed`, `invalid jwt`
  1282‚Üí- `authentication failed`, `auth token invalid`
  1283‚Üí- `login required`, `user not authenticated`, `not authenticated`
  1284‚Üí- `session expired`, `account locked`, `expired token`, `invalid or expired`
  1285‚Üí- `proxy returned 400`, `proxy returned 401`, `proxy returned 403`
  1286‚Üí
  1287‚ÜíAlso: `data.validToken === false && data.status === "fail"` ‚Üí true.
  1288‚Üí
  1289‚ÜíOn confirmed expiry ‚Üí `code:"TOKEN_EXPIRED", needsReauth:true`.
  1290‚Üí

  1322‚Üí

  1336‚Üí## 7. End-to-end flow summary
  1337‚Üí
  1338‚Üí1. **User links account:** OTPDialog ‚Üí `send-otp` ‚Üí `verify-otp` ‚Üí `FantasyAccount.authToken` saved to `localStorage.tg_multi_accounts[platform][0]`.
  1339‚Üí2. **User opens transfer screen:** `TeamTransferScreen` calls `fetchExistingTeams()` ‚Üí `list-of-teams` ‚Üí `existingTeams[]` populated, `presentTeamCount` and `newSlots` calculated (`maxTeams` = 11 for dream11, 40 for others).
  1340‚Üí3. **User picks mode:** "all" / "newSlotsOnly" / "custom" (or replace-selected).
  1341‚Üí4. **`buildTransferPayloads(mode)`** builds one payload per team. Captain/VC = fantasy IDs as numbers, players = `number[]` of fantasy IDs (‚â•11), `type: "edit"|"new"`, `id: existingTeam.team_id` for edits, `userToken: localStorage.user_token` (optional).
  1342‚Üí5. **`executeTransferBatch(payloads)`** sends each payload (with `skipPreVerify: true`) to `/api/fantasy/transfer`.
  1343‚Üí6. **`transfer/route.ts`** validates, resolves Bearer via `resolveBearerToken(userToken)`, builds the final backend payload (drops `type` and `userToken`), POSTs to the platform's `add-team`/`edit-team` endpoint chain.
  1344‚Üí7. **Backend response:** Only `data.status === "success"` counts as success.
  1345‚Üí8. **On token-expiry error:** Frontend tries `auth-verify` to refresh; if backend returns `updateUiToken + authToken`, updates localStorage and retries the failed team.
  1346‚Üí9. **On success:** Toast + refetch existing teams after 1 s.

  1350‚Üí## 8. Key gotchas (must-know for any future refactor)
  1351‚Üí
  1352‚Üí1. **Captain/VC MUST be numbers** (the player's fantasy ID, not array index). Backend rejects strings.
  1353‚Üí2. **authToken MUST be the FULL raw token** (JSON-wrapped if backend returned JSON). Extracting just `accessToken` causes "Something Went Wrong!".
  1354‚Üí3. **`data.status === "success"` is the ONLY success criterion.** HTTP 201 + `{status:"fail"}` is a FAILURE.
  1355‚Üí4. **Players array must have ‚â• 11 numeric IDs.** Teams with fewer are skipped silently at frontend.
  1356‚Üí5. **Dream11 has TWO add endpoints** (`/api/fantasy/add-team` first, then `/api/classic/dream11/addteam` as fallback). Others have only one.
  1357‚Üí6. **list-of-teams runs endpoints in parallel** (`Promise.any`) ‚Äî first success wins.
  1358‚Üí7. **`Authorization: Bearer` is OPTIONAL** on the main transfer path (bypass mode). It is REQUIRED on `list-of-teams` and `auth-verify` (always sent, even if empty).
  1359‚Üí8. **Edit‚Üíadd fallback** (lines 561‚Äì650) deletes `payload.id`, sets `payload.type = "new"`, and forces `Authorization: Bearer ${bearerToken}` (even if empty ‚Äî a quirk).
  1360‚Üí9. **my11circle requires 3 extra fields:** `my11circleChallenge`, `my11circleUserId`, `my11circleMobile`. Without these, transfers fail.
  1361‚Üí10. **`myteam11` uses `Loginauthtoken`** (capital L) in verify-otp ‚Äî original APK quirk preserved.
  1362‚Üí11. **`sportIndex`:** 0=cricket, 1=football, 2=basketball, 3=kabaddi. Defaults to 0.
  1363‚Üí12. **`maxTeams`:** dream11 = 11, my11circle/jumbo = 40.
  1364‚Üí13. **Frontend batch size = 5** with per-platform delays (200/800/500 ms). Backend per-endpoint timeout = 15 s (main) or 30 s (fallback).
  1365‚Üí14. **Token refresh:** Only attempted ONCE per team. If `tryRefreshToken()` fails, all remaining teams in the batch are failed.
  1366‚Üí15. **`skipPreVerify: true`** is sent by frontend on every transfer request ‚Äî pre-verification is bypassed for speed. Token expiry is detected reactively.
